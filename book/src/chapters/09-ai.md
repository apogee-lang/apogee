# Apogee and AI

## Why AI Needs a Better Target Language

AI writes code now. That's not a prediction — it's Tuesday. Copilots, Claude, GPT, and dozens of other tools generate code millions of times per day. And overwhelmingly, they generate Python and JavaScript.

This is a problem.

Python and JavaScript have no compile step. No null safety. No constraint enforcement. When AI generates code with a bug, you find out at runtime — maybe in production. The feedback loop is:

1. AI generates code
2. You run it
3. It crashes
4. You read a stack trace
5. You ask AI to fix it
6. Repeat

Apogee collapses this loop. When AI generates Apogee code with a bug, the *compiler* catches it immediately — before the code ever runs. The feedback loop becomes:

1. AI generates code
2. The compiler says "line 7: null safety violation"
3. AI fixes line 7
4. It compiles. It works.

The compiler is the fastest, cheapest, most reliable code reviewer you'll ever have.

## How to Use Claude to Write Apogee

Here's a practical guide to using AI assistants to write Apogee code effectively.

### The System Prompt

Add this to your Claude system prompt (or project instructions) to get Apogee code by default:

```
You are writing code in Apogee, a statically-typed language that compiles
to Python. Key syntax:

- Functions: fn name(param: Type) -> ReturnType { body }
- Types: type Name { field: Type }
- Nullable: String? means nullable, String means never null
- Constraints: Int where value >= 0
- Intent: @intent("describe purpose") before functions
- Queries: from list where it.field == value
- Concurrency: spawn { task1(); task2() }
- String interpolation: "\(variable)"
- Comments: // line, /* block */
- No semicolons needed. Last expression is return value.

Always add @intent annotations to functions. Use constraint types
for domain values (ages, prices, ports, etc.). Prefer from/where
over manual loops for filtering.
```

With this prompt, Claude generates idiomatic Apogee — including `@intent` annotations, constraint types, and query expressions.

### Example: Asking Claude to Build a Feature

**Prompt:** "Write an Apogee function that takes a list of orders and returns the total revenue for orders in a given region."

**Claude generates:**

```
type Order {
  id: Int where id > 0
  amount: Float where amount >= 0
  region: String
}

@intent("calculate total revenue for all orders in the specified region")
fn revenue_by_region(orders: [Order], region: String) -> Float {
  let regional = from orders where it.region == region
  let total = 0.0
  for o in regional {
    // total = total + o.amount
  }
  total
}

let orders = [
  Order { id: 1, amount: 99.99, region: "North" },
  Order { id: 2, amount: 149.50, region: "South" },
  Order { id: 3, amount: 75.00, region: "North" },
  Order { id: 4, amount: 200.00, region: "South" }
]

print(revenue_by_region(orders, "North"))
```

Notice what Claude got right automatically:
- `@intent` annotation
- Constraint types on `id` and `amount`
- `from ... where` for filtering
- Clean struct literals

### The Edit-Compile-Fix Loop

The most powerful AI+Apogee workflow:

1. **Ask AI** to generate a function
2. **Run `apogee check`** on the result
3. If there are errors, **paste the error back to AI** and ask for a fix
4. Repeat until it compiles
5. **Run `apogee run`** to verify behavior

This loop is fast because Apogee's error messages are specific:

```
// AI generates:
let user = find_user(42)
print(user.name)

// Compiler says:
// [2:7] Type error: Cannot access .name on User? — value might be null
//   Suggestion: Use 'user?.name' or check for null first

// You paste the error back to AI. AI fixes it:
let user = find_user(42)
if user != null {
  print(user.name)
} else {
  print("User not found")
}
```

The compiler error is so specific that the AI can fix it in one shot. Compare this to Python, where you'd get an `AttributeError` stack trace at runtime — much harder for AI to diagnose because it lacks context about *why* the value was null.

## @intent as AI-Verifiable Contracts

`@intent` annotations serve a dual purpose:

### For Humans

They document what a function is supposed to do. A developer reading the code understands the intent without reading the implementation.

### For AI

They provide a machine-readable specification that AI tools can verify:

```
@intent("sort users by age ascending, do not modify original list")
fn sorted_users(users: [User]) -> [User] {
  // If AI verification detects this doesn't sort,
  // it can flag a warning: "implementation may not match intent"
  from users where true
}
```

Today, this is aspirational — the `@intent` compiles to a docstring. But the annotation is *already there*, ready for verification tools. When those tools arrive, every Apogee codebase that uses `@intent` will benefit immediately.

### Writing Intents That AI Can Verify

Good intents are specific enough to check:

```
// Verifiable:
@intent("return the count of active users, always >= 0")
fn active_count(users: [User]) -> Int {
  len(from users where it.active == 1)
}

// Verifiable:
@intent("never return a price less than zero")
fn discounted_price(price: Float, discount: Float) -> Float {
  let result = price * (1.0 - discount)
  if result < 0.0 { 0.0 } else { result }
}

// Verifiable:
@intent("return the input list reversed, same length as input")
fn reversed(items: [Int]) -> [Int] {
  items  // BUG: this doesn't reverse!
  // A future AI verifier would flag: "output appears identical to input,
  // but intent specifies reversal"
}
```

The key properties that make intents verifiable:
- **Quantitative claims** ("always >= 0", "same length as input")
- **Behavioral descriptions** ("reversed", "sorted ascending")
- **Null/safety guarantees** ("never return null", "never throw")

## How Apogee Makes AI Errors Visible

Every advantage of Apogee over Python is amplified when AI is writing the code:

| Error Type | Python | Apogee |
|-----------|--------|--------|
| Null access | Runtime crash | Compile error |
| Wrong field name | Runtime AttributeError | Compile error |
| Invalid constraint | No checking | Compile error |
| Missing struct field | Runtime error | Compile error |
| Type mismatch | Runtime TypeError | Compile error |

AI makes these mistakes constantly. In Python, they ship to production. In Apogee, the compiler catches them before the code runs.

```
// AI generates (with a bug):
type Config {
  host: String
  port: Int where port > 0
}

let c = Config { host: "localhost", port: 0 }
// COMPILE ERROR: Constraint violation: port = 0 does not satisfy port > 0
```

The AI made a mistake (port 0 is invalid). Python would accept it silently. Apogee catches it immediately.

## AI-Assisted Refactoring

Because Apogee has types and intents, AI refactoring is safer:

```
// Original:
@intent("check if user can access the admin panel")
fn can_access_admin(user: User) -> Bool {
  user.role == "admin" && user.active == 1
}

// Ask AI: "Add a check for email verification"
// AI generates:
@intent("check if user can access the admin panel, requires verified email")
fn can_access_admin(user: User) -> Bool {
  user.role == "admin" && user.active == 1 && user.verified == 1
}
```

The `@intent` was updated to match the new behavior. If the AI forgot to update the intent, a future verification tool would flag the mismatch.

> **vs Python**
>
> When AI refactors Python code, there's no type checker to verify the changes are safe. No intent annotation to check for consistency. You review the diff manually and hope you catch everything.
>
> With Apogee, the compiler is the first reviewer. It catches type errors, constraint violations, and null safety issues. The `@intent` annotation catches intent drift. The AI + compiler combination is more reliable than AI alone.

## Common Mistakes

**Not giving AI the system prompt.** Without the Apogee syntax guide, AI will generate Python. Always include the system prompt.

**Not pasting compiler errors back to AI.** Apogee's errors are specific enough that AI can usually fix them in one try. Don't fix them manually — let the AI learn from the error.

**Ignoring @intent on AI-generated functions.** If AI generates a function without `@intent`, add one. It's the single best thing you can do for future maintainability — human or AI.
